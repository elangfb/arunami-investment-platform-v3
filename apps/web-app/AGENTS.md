## Commands
Run from the repo root.

- `pnpm dev` — Next dev on `:3000` through Nx.
- `pnpm typecheck` — Next typegen + TypeScript.
- `pnpm lint` — ESLint.
- `pnpm test:unit` — hermetic `node:test`/tsx tests; no DB.
- `pnpm test:integration` — DB-backed tests against `mizan_test`; script refuses non-test DBs. Use for repo writes/transitions.
- `pnpm build` — production build.
- `pnpm emu` — Firebase Auth emulator for local login.
- `pnpm seed:emu` — provision demo login accounts in the running emulator (and drop smoke-* cruft); roster is `src/lib/seed-data/demo-logins.ts`. Run after `pnpm emu` + `pnpm seed`.
- `pnpm gen:dummy` — regenerate end-to-end test fixtures under `resources/dummy-data/<persona>/` (cheat-sheets `application.txt`/`nasabah.txt` + a real PDF per required doc) from `src/lib/dummy-data/personas.ts`. PDFs OCR back the gate values under a real provider; the default `stub` provider ignores bytes (fabricates from app data).

## App Facts
- App path: `apps/web-app`.
- Stack: Next 16 App Router, React 19, Tailwind 4, shadcn + Base UI, Prisma 7/Postgres, Firebase Auth, Google APIs.
- Runtime data is Postgres via `server/repo/*`; `src/lib/seed-data/` is seed source only.
- Seed split: `seed-config.ts` is prod-safe factory defaults; `seed-dummy.ts` is dev-only demo data; prod runs `pnpm seed:config` after migrate. Seed is **non-destructive and idempotent**: users/desks upsert by unique key; seeded apps scoped-replace only their own ids — real users/sessions/non-seeded apps SURVIVE a re-seed. Command: `cd apps/web-app && set -a; . .env.local; set +a; pnpm exec tsx prisma/seed.ts`. FOS-035/036/037 are state-coverage seed apps (mid-disbursement · ready-to-release · awaiting-risk-reco) useful for testing distinct detail-page states.

- Env file: `apps/web-app/.env.local`; Prisma config: root `prisma.config.ts`.
- Migrations: `set -a; . apps/web-app/.env.local; set +a; ./apps/web-app/node_modules/.bin/prisma migrate dev --name <name>`.
- **If `migrate dev` aborts on drift** (a modified-after-applied migration, or an out-of-band schema change in the dev `mizan` DB — common locally), do NOT `migrate reset` (data loss). Author the migration without touching dev state: deploy history into a throwaway DB (`DATABASE_URL=…/mizan_shadow prisma migrate deploy`), then `DATABASE_URL=…/mizan_shadow prisma migrate diff --from-config-datasource --to-schema apps/web-app/prisma/schema.prisma --script` to emit the delta SQL, hand-write it into `prisma/migrations/<ts>_<name>/migration.sql`, and `prisma migrate deploy` (no drift check) to apply to **both** `mizan` + `mizan_test`. - After migrations or Prisma dependency changes: `prisma generate`, fully stop Next dev, verify `:3000` is free, then restart.
- Deployment runbook: `docs/guides/deployment.md`. Preserve Docker rules: pnpm build-script workaround, standalone Prisma WASM copy, build-time `NEXT_PUBLIC_*`, and lazy server secret init.

### Documents, OCR, AI, and templates
- Document files are real bytes: byte-validated, SHA-256 tracked, served only through the authed proxy route. Never trust `file.type`, filenames, or public/presigned URLs.
- OCR providers go through `server/ocr/` via env-only `OCR_PROVIDER`: `stub` (offline default/dev/test/CI), `documentai` (production Google Document AI; see `docs/guides/document-ai-ocr.md`), `gemini` (interim generative cloud). External OCR is PII egress and remains a human-confirmed suggestion. Add providers by implementing `OcrProvider` — no call-site changes. KTP already OCRs for NIK; do **not** run `extractFullText` on KTP (double-call risk).
- OCR confirmation is registry-driven: `confirmExtractedFieldAction(appId, fieldPath, override?, resolution?)` confirms any suggested field, gated by owner desk/stage via `assertCanWorkDesk`. Plain confirm only flips provenance; `override` is identity-only (NIK). Gating values (Kol, income, appraisal) stay in their dedicated editors. `ocrBlockers(app, desk)` is desk-scoped; the Data-nav badge may add conservative required-but-empty signals but must not count optional blanks.
- **OCR cross-check (Batch 6): OCR never blind-overwrites a blessed value.** When OCR re-reads a field, `reconcileExtraction(existing, provenance, ocrValue)` (`lib/extraction-registry.ts`) decides: **fill** (empty/unblessed → write suggestion, legacy behavior), **match** (blessed + agrees → no-op + clear any stale conflict), **mismatch** (blessed `human_entered`/`ocr_confirmed`/`ocr_overridden` + differs → record, KEEP the Mizan value). Conflicts persist in the `extractionMismatches` JSON column (`Record<fieldPath, ExtractionMismatch>`); both the gate-suggestion path AND the NIK path go through `applyExtractionCandidate`. The owner resolves via `confirmExtractedFieldAction(…, undefined, 'keep'|'accept')` — `accept` writes the OCR value back as a fresh `ocr_suggested` (gating re-enters confirm+recompute, never blind credit); `planMismatchResolution` keeps raw **NIK out of the audit ledger** (PII), records only gating numeric deltas. **A NIK mismatch BLOCKS the MUAP→Risk handoff** (`muapToRiskBlockers` — relocated from the 1→2 advance by P3-B 2026.06.11; the Inisiasi 1→2/2→3 advances are now FREE, all intake gates sit at MUAP→Risk); gating mismatches are advisory and are NOT `ocrBlockers` — they only raise `dataAttentionCount`. ### Runtime and auth gotchas
- Auth is Firebase session-cookie based. For local role testing, use Auth emulator + `SUPERADMIN_EMAILS`, then impersonate desks; never restore localStorage switching. Demo emails in `src/lib/seed-data/demo-logins.ts` must match the seeded roster (`ensureUser` links by email).
- Keep context provider values and Base UI `Select` item arrays stable when effects depend on them.
- Server logs go through `server/log.ts`; never log PII.
- Runtime Google/Gemini calls go through `withRetry()`.

## API & Security
- Every `/api` route starts with `verifySession()`.
- Mutating, PII, or external-egress routes also need a desk/participation gate and `rateLimit`.
- On 500s: log with `log.error(...)`; return a generic Bahasa message.
- Treat auth, audit, PII/OCR, document storage, MUAP/RSK, hard gates, finance calculations, and stage transitions as high-risk.
