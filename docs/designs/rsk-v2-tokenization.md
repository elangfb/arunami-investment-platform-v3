## Conventions

Same A/B/C/D/E/F categorization and `snake_case` rules as MUAP. RSK-specific:

- **`rsk_*` prefix** for all RSK-owned tokens (fact + categorical).
- **`ai_rsk_*` prefix** for RSK-owned AI narratives. (`ai_*` without `rsk_` = MUAP-shared AI tokens reused by RSK.)
- **Fixed-slot row caps** for variable-row tables (covenants, monitoring, red flags) — sized from the largest realistic case, not the template demo. Final caps: covenants 10, monitoring 10, red-flag 8.
- **Policy reads, not fact fills.** SCCR minimum, BMPD/BMPK limits, DSCRi threshold are read from versioned `RiskPolicyVersion` at fill time, **not** placed as facts. (Aligns with the versioned-config decision in `admin-config-layer.md`.)

## Reuse tiers (R1–R5)

Every RSK cell was classified by reuse provenance:

- **R1 [muap]** — reuses an existing MUAP canonical token (~80 cells).
- **R2 [new]** — net-new RSK-owned fact / categorical.
- **R3 [new]** — net-new RSK-owned AI narrative.
- **R4 [computed]** — derived at fill time (e.g. SCCR ratio = Σ likuidasi / plafond).
- **R5 [policy]** — read from versioned config (BMPD limit, SCCR min, DSCRi threshold).

The build-time registry must enforce: an R1 claim that doesn't resolve to a real MUAP token → **fail loud**. (See Pitfall 3.)

## 5C+2S — the Hijra literal

Dimensions in PDF order (not academic 5C+1S):

`CHARACTER` · `CAPACITY` · `CAPITAL` · `COLLATERAL` · `CONDITION` · `SHARIA COMPLIANCE` · `SHARIA STRUCTURING`

- **Sharia Compliance** ≈ halal-objek + DPS review.
- **Sharia Structuring** ≈ akad fitness + tidak ada riba/gharar.
- The order matters — Compliance comes **before** Structuring on the rendered Doc. The app-side mapping lives in `app.analysis.*`; see `apps/web-app/AGENTS.md` and `docs/CURRENT-STATE.md` for current build/audit caveats.

## Pitfalls

1. **"95 brackets" ≠ 95 fillable cells.** Rev-1 baseline counted distinct bracket *strings*; the PDF has ~115 distinct fillable *cells* once `[X]` repetitions collapse across financial tables. Sigblock underscores were also missed in the baseline (4 vs the real 13). Always count cells in the PDF, not bracket occurrences in the text export.
2. **Multi-row tables are not always homogeneous.** §IV.1 Daftar Agunan reads as "5 collateral rows" but slots 3/4/5 are fixed-role (cash collateral / cessie / PG) with different shapes — not five property rows. Same for §VI.1 (13 rows span across MUAP T42 + T44 + T45). Inspect each row before assuming a template.
3. **The MUAP-promotion temptation (THE single most important lesson).** Three separate agents tried to "promote" RSK-needed fields up to MUAP rather than own them on the RSK side. The rule, encoded in `apps/web-app/AGENTS.md` + the playbook Pitfall 11: **Hijra templates are literal source of truth. If RSK needs a field MUAP doesn't have, RSK owns it (`rsk_*`).** The build-time registry must refuse any R1 `[muap]` claim that doesn't resolve to an existing MUAP token. Rev 2.1 reclassified 16 such invented promotions; rev 2 had silently introduced them.
4. **Wishful R1 names.** Rev 1.2 used 95 R1 reuse claims; only ~5 used the correct MUAP canonical name. The other 90 were retargeted in rev 2 (e.g. `nasabah_nama` → `nama_perusahaan`; `akta_pendirian_no` → `no_akta_pendirian`; `nib` → `no_nib`). Lesson: always grep the MUAP canonical registry; never type a token name from memory.
5. **Sibling-rename build-order dependency.** RSK §V.4 reuses `baki_debet_slik_*`, which only exists after MUAP rev 2.2's rename of `plafond_slik_*`. RSK builds depend on MUAP rev 2.2 being shipped first. When tokenizing sibling templates, ship the dependency rename in the same commit.
6. **Fixed-row caps from the template demo are usually too small.** Monitoring demo had 7 rows but the rev-1.2 cap was 6; red-flag was capped at 6 vs Q3's 8. Size caps from realistic worst case, not template demo.

## What the registry must enforce at build time

The unified token registry (`apps/web-app/src/lib/templates/tokens.ts` — T1 task, see `muap-v2-next-steps.md`) gates these invariants for both MUAP and RSK:

1. Exactly one prefix per kind (`ai_*` only; no `ain_*`, no bare `uraian_*`).
2. Every R1 `[muap]` claim resolves to a live MUAP-side token, else **fail loud**.
3. AI may only write `ai_*` slots (deterministic fills everywhere else).
4. Reuse > duplicate — same concept across sections shares one NamedRange.
5. Fixed-slot caps match realistic worst case, not template demo.
