# pi worker — discrepancy & improvement log

> Purpose: every time a pi worker's result/output **doesn't match what the
> orchestrator would have produced** (wrong answer, missed blast radius, drift
> from the prompt, false "done", stale citations, scope creep, format
> violations), record it here. Over time these entries feed improvements to
> how we phrase tasks and any future pi prompt overlay — so pi can do the work the
> way the orchestrator would.

## How to log an entry
One entry per discrepancy. Keep it terse and actionable:

```
### <date> · <task-id> · <one-line title>
- **Tier/model:** low | medium
- **Task (verbatim or gist):** …
- **pi output:** what it returned/did
- **Discrepancy:** how it differed from the expected/correct result
- **Root cause (hypothesis):** prompt ambiguity? model limitation? missing context?
- **Fix applied / takeaway:** prompt change, tier bump, or durable rule to add
```

### 2026-05-24 · bravo-8 · false typecheck-failure attribution
- **pi output:** bravo reported "pnpm typecheck fails in out-of-scope DocumentsTab".
- **Discrepancy:** it was a transient mid-edit state (charlie editing concurrently);
  final typecheck passed. bravo surfaced a cross-worker race as a failure.
- **Takeaway:** when fanning parallel workers over sibling files, a shared typecheck
  mid-run can read another worker's half-written file. Have workers typecheck only
  their own file's island, or run the project typecheck once after ALL workers finish.

### 2026-05-25 · alfa-2 · `as any`/`@ts-ignore` to silence repo-boundary type errors
- **pi output (--medium):** cut meetings.ts to the new join tables — LOGIC correct —
  but littered the file with `(prisma.komiteMeeting as any)`, `(prisma as any)`,
  a `@ts-ignore`, and a hand-rolled `MeetingRow` Omit-override, claiming "generated
  Prisma client may lag the migrated schema".
- **Discrepancy:** the client had just been regenerated by `migrate dev`; the types
  were current. The REAL typecheck error was unrelated (old required columns
  agendaAppIds/attendeeUserIds still in schema → CreateInput demanded them). pi
  papered over a real, fixable type error with `any` in a safety-critical boundary
  instead of reporting it.
- **Root cause:** when a worker sees a type error it can't immediately explain, low/med
  tends to cast to `any` rather than BLOCK or root-cause. In the repo/serialize boundary
  (CLAUDE.md "safety-critical") that hides real schema/contract bugs.
- **Fix applied / takeaway:** orchestrator rewrote meetings.ts with zero casts
  (`Prisma.KomiteMeetingGetPayload` + `satisfies KomiteMeetingInclude`); typecheck then
  surfaced the true cause (drop the legacy columns). Add to repo-boundary task prompts:
  "NEVER use `as any`/`@ts-ignore` to silence a type error — if types seem stale, run
  `prisma generate` and report the exact error as BLOCKED; do not cast it away."

### 2026-05-25 · bravo-3 · "tests pass" via tsx runner missed tsc-strict error
- **pi output (--medium):** wrote conversation.itest.ts (correct, faithful to spec),
  verified with `pnpm test:integration` (tsx) → green, reported "notes: none".
- **Discrepancy:** `pnpm typecheck` (tsc --noEmit) then FAILED — it passed
  `app.version` (domain type `number | undefined`) into a `number` param at 5 sites.
  The integration runner (tsx) doesn't enforce strict tsc, so "tests pass" hid it.
- **Takeaway:** for tasks whose SUCCESS is a test run, the worker's green from the
  tsx-based runner is necessary but NOT sufficient — add "also run `pnpm typecheck`"
  to the SUCCESS line whenever the change adds/edits a .ts file the project tsc covers.
